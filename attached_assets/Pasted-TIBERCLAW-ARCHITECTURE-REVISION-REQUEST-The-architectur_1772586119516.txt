1. Doctrine Module Output Contract

All Doctrine modules must return a standardized evaluation object so downstream systems (GM layer, LLM gateway, strategy endpoints) can consume them reliably.

Add a shared Doctrine response schema to the architecture doc.

Required fields:

* module
* entity_type (team | player | league)
* entity_id
* evaluation_score (0–1 normalized)
* confidence (0–1)
* contributing_signals[]
* reasoning
* generated_at

This schema will be used by all Doctrine modules:

* team_window_detection
* asset_insulation_model
* positional_aging_curves
* league_market_model
* roster_construction_heuristics

Do not allow modules to invent custom output formats.

3. League Snapshot Lifecycle

Define when league snapshots are created.

Add snapshot triggers:

* league_import → create baseline snapshot
* weekly_rollover → snapshot after final MNF game
* transaction_event → optional snapshot on trades/waivers
* manual_refresh → triggered by `/api/league-sync/sync`

Also define retention rules:

* weekly snapshots retained indefinitely
* transaction snapshots retained for 90 days

This prevents agents from over-querying the Sleeper API and ensures consistent league state.

4. LLM Boundary Rule

Add an explicit rule to the architecture:

LLMs must only receive Doctrine outputs.

LLMs must not receive raw Tiber engine metrics or perform strategic analysis.

Their role is narrative explanation only. Strategic reasoning must occur entirely within Doctrine modules.

Doctrine modules will be implemented in the Node/TypeScript service, not in Python.

Python remains responsible for ELT pipelines and heavy statistical modeling. Doctrine logic belongs in the API layer for low-latency decision evaluation.

server/doctrine/

Each module should live independently within this directory.
